# Заявка на покупку

## Обзор

Веб-приложение для управления заявками на покупку. Пользователи создают и отправляют
заявки на покупку, а проверяющие рассматривают их: подтверждают или отклоняют. Список
заявок поддерживает сортировку по ключевым полям (дата создания, статус, сумма, автор).
Доступ к функциям разграничен системой авторизации по ролям.

## Основные возможности

- **Авторизация и аутентификация**
  - Регистрация и вход по email и паролю.
  - Аутентификация через JWT (access-токен, при необходимости — refresh-токен).
  - Разграничение доступа по ролям: заявитель (создаёт заявки) и проверяющий
    (рассматривает заявки).
- **Приём заявок**
  - Создание заявки на покупку (наименование, количество, сумма, комментарий).
  - Просмотр заявителем своих заявок и их текущего статуса.
  - Валидация входных данных на стороне сервера.
- **Проверка заявок с сортировкой**
  - Список всех заявок для проверяющего.
  - Подтверждение или отклонение заявки с фиксацией результата.
  - Сортировка списка по полям: дата создания, статус, сумма, автор
    (по возрастанию и убыванию).
  - Пагинация списка заявок.

## Технологический стек

- **Язык программирования:** TypeScript
- **Backend-фреймворк:** NestJS (Node.js)
- **Frontend-фреймворк:** Vue 3 на TypeScript (Composition API, `<script setup lang="ts">`, Vite)
- **База данных:** PostgreSQL
- **ORM:** Prisma
- **Аутентификация:** JWT (@nestjs/jwt, Passport)
- **Валидация:** class-validator / class-transformer (backend)

## Архитектурные заметки

- **Backend (NestJS):** доменные модули — `auth`, `requests` (заявки), `users`.
  Бизнес-логика — в сервисах, HTTP-контракт — в контроллерах и DTO. Доступ
  разграничивается через Guards (JWT + роли). Работа с БД инкапсулирована в
  Prisma-сервисе и репозиториях/сервисах модулей.
- **Frontend (Vue 3 + TypeScript):** слоистая структура — API-клиенты, сторы (Pinia)
  для состояния и бизнес-логики, компоненты страниц и переиспользуемые UI-компоненты.
  Весь код на TypeScript (`<script setup lang="ts">`); контракты компонентов
  (`props`/`emits`) выносятся в отдельные файлы типов.
- **Данные:** ключевые сущности — `User` (с ролью) и `PurchaseRequest` (с полем
  статуса и ссылкой на автора). Сортировка и пагинация выполняются на уровне БД
  через Prisma (`orderBy`, `skip`/`take`).
- **Разделение слоёв:** типы и контракты централизованы в отведённых местах слоя,
  бизнес-логика не размещается в контроллерах и компонентах.

## Нефункциональные требования

- **Логирование:** уровень настраивается через `LOG_LEVEL`; структурированные логи.
- **Обработка ошибок:** единый формат ответов об ошибках (exception filters в NestJS).
- **Безопасность:** хеширование паролей (bcrypt/argon2), проверка ролей на каждом
  защищённом эндпоинте, валидация всех входных данных, CORS и секреты через
  переменные окружения (`DATABASE_URL`, `JWT_SECRET`).
- **Конфигурация:** параметры окружения через `.env` (подключение к БД, секреты JWT,
  порт, уровень логирования).
- **Сидеры БД:** отдельный скрипт наполнения PostgreSQL начальными данными
  (Prisma seed, `prisma/seed.ts`, запуск через `prisma db seed`) — тестовые
  пользователи с ролями (заявитель, проверяющий) и демонстрационные заявки на
  покупку с разными статусами для проверки списка, сортировки и пагинации.
